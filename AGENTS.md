## CONVENTIONS

**Git Workflow**:
- Branch from `main`: `git checkout main && git pull && git checkout -b feat/xxx`
- Commit format: `[з±їеЮЛ] жППињ∞ (#PRеПЈ)` - types: `fix`, `feat`, `mod`, `dev`, `conf`
- вЭМ Never commit on merged feature branches
- вЭМ Never force push to main

**Version Management**:
- Version ONLY in `modpack.toml` - CI auto-updates other configs
- Test builds: `test-*` branches вЖТ version appended with `-test-build-{n}`
- Releases: `release*` branches

**Mod Management (Packwiz)**:
- `mods/`, `resourcepacks/`, `shaderpacks/` contain `.pw.toml` metadata; no `mods/*.jar` files are tracked
- CF-restricted/custom JARs/zip live in `packwiz-files/{mods,resourcepacks,shaderpacks}/`
- Any committed mod JAR, including CDC dev builds and CF/manual-download fallbacks, must live in `packwiz-files/mods/` with matching `mods/*.pw.toml` raw URL metadata
- Shaderpack files containing `Clrwl` are generated by mods and must not be tracked in `.pw.toml` or `packwiz-files`
- `.pw.toml` for packwiz-files assets reference GitHub raw URLs generated by `scripts/update-packwiz-meta.ps1`; local metadata defaults to the current ref, and release CI pins URLs to `github.sha` to avoid stale branch payloads
- CF-hosted assets kept in `packwiz-files` may include `[release.curseforge]` so release workflows can emit manifest metadata while dev sync uses local payloads
- To add/update/remove mods/resourcepacks/shaderpacks: use `scripts/update-packwiz-meta.ps1 -Category ...` (NOT manual asset placement)
- To sync all mod JARs locally for development: use `scripts/sync-packwiz-assets.ps1`
- `pack.toml`/`index.toml` are generated from `modpack.toml`; don't commit them
- `CDC-mod-src/` is a git submodule and must stay out of Packwiz artifacts because packages ship pack files, not Java source trees
- For CDC artifact updates, use matching CF metadata when published; otherwise put the dev JAR in `packwiz-files/mods/`, update `mods/create-delight-core.pw.toml`, and align the `CDC-mod-src/` submodule pointer

## ANTI-PATTERNS

- вЭМ `rm -rf`, `del /S /Q` on config/, kubejs/, mods/, hotai/, PCL/
- вЭМ Batch delete `*.json`, `*.snbt` in core dirs
- вЭМ Overwrite `ModList*.md`
- вЭМ `e.remove()` or `e.removeById()` - use `remove_recipes_id(e, [...])`
- вЭМ Duplicate version in other files
- вЭМ Treat runtime dirs (`logs/`, `crash-reports/`, `saves/`, `screenshots/`, `simplebackups/`, `tmp-*`) as source

## KNOWLEDGE BASE MAINTENANCE

These rules ensure this knowledge base stays effective. Violating them degrades agent performance.

- **Root AGENTS.md вЙ§150 lines** вАФ If over limit, prune stale entries or move content to subdirectory files
- **Subdirectory AGENTS.md вЙ§80 lines** вАФ Keep focused on that domain only
- **No duplication** вАФ Each fact exists in exactly ONE file. Others reference it. Duplicate information = conflicting information
- **Concise > verbose** вАФ One sentence per fact. No prose. Agents ignore buried instructions
- **Include the Why** вАФ Non-obvious rules must explain the reason (agents follow rules better when they understand the failure mode)
- **Stale > harmful** вАФ Outdated instructions are worse than no instructions. Update when architecture changes, prune aggressively
- **Iterate, don't upfront** вАФ Add rules only when agent makes a repeated mistake. Remove rules agent always follows correctly
- **Lessons go to `lessons-learned.md`** вАФ Never inline long historical entries in AGENTS.md
- **After fixing a non-obvious bug** вАФ Add entry to `lessons-learned.md` (use `/knowledge-check` skill for guidance)
- **Validation** вАФ Codex `Stop` hook runs `scripts/validate-knowledge-base.ps1` to catch line-limit and stale-path failures
- **Candidate report** вАФ Codex `Stop` hook writes `tmp-opencode/knowledge-candidate-report.md`; it is advisory only and never edits knowledge files
- **Process notes** вАФ When a task hits a non-obvious failure or workaround, append a temporary note with `scripts/add-knowledge-note.ps1`; the candidate report routes it for `/knowledge-check`
- **Candidate decision** вАФ Apply/reject candidate reports with `scripts/resolve-knowledge-candidate.ps1`; process-note candidates need user acceptance unless knowledge maintenance was explicitly requested
